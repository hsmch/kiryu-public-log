## 1. プロジェクトの原点と問い

### なぜ KPL を始めたか

桐生市では近年、市政に関わる重大な問題が相次いだ。

- **生活保護の支給不正問題**: 市の福祉行政における支給手続きの適正性が問われた
- **市役所新庁舎建設の入札妨害事件**: 大型公共事業における入札プロセスの公正性が問題となった

これらの事案は、市民が市政の運営実態を把握し、今一度省みる必要があることを示している。

しかし現実には、議会や市政の情報は「公開されているが、見えにくい」状態にある。桐生市議会の公式サイトには議案や投票記録が掲載されているが、PDFや分散したページに埋もれており、市民が全体像を把握することは極めて困難である。

KPL は以下の仮説に基づいて始まった:

> **市民が市政に前向きに関わるためには、まず実態を把握できる環境が必要である。
> 情報へのアクセシビリティを高めることが、市民参加の一歩目になる。**

今後の桐生市図書館の建て替えをはじめとする大型プロジェクトにおいて、市民が情報に基づいた判断と参加ができるようにすること。それがKPLの目指すところである。

### 本レポートで検証する問い

1. **サイトとしての方向性**: 「中立的なアーカイブ」のままでよいか、「市民参加促進プラットフォーム」まで踏み込むべきか
2. **担保すべき情報**: 詳細度・網羅性・解説・閲覧性のバランスをどう取るか
3. **気づきと行動の仕組み**: 情報提供の先にある「気づき」と「行動」をどう設計するか
4. **このアプローチ自体の妥当性**: そもそもこの方向で検証を進めることは正しいか

## 2. 検証アプローチの妥当性

### 結論: 妥当である

本検証は以下の3つの専門的視点から並行して分析を行った:

| 視点 | 目的 |
|------|------|
| シビックテック事例研究 | 国内外の類似プロジェクトの成功パターンと失敗要因を把握する |
| 現状ギャップ分析 | KPLの現コンテンツを精査し、動機との乖離を明らかにする |
| プロダクト戦略分析 | ユーザーセグメント・ファネル設計・スコープの妥当性を検討する |

このアプローチが妥当である理由:

1. **動機と現状の間にギャップがある**: プロジェクトの動機（不正問題・入札妨害→市民が市政を省みる必要）と、現サイトの機能（データアーカイブ）の間には明確なギャップが存在する。このギャップを認識し、埋める戦略を考えることは正しいステップである
2. **先例が少ない**: PMO（Parliamentary Monitoring Organization）の国際調査では、日本は議会モニタリング組織が「非常に限定的」と分類されている。人口10-15万人規模の地方自治体を対象にした議会モニタリングサイトは日本にほぼ存在しない。先例が少ないからこそ、海外事例や戦略フレームワークからの逆算が有効
3. **個人運営という制約**: リソースが限られているため、「何をやるか」以上に「何をやらないか」の判断が重要。戦略的な優先順位づけなしに施策を積み上げると、持続可能性を損なう

## 3. 現状分析: KPLが今どこにいるか

### 3.1 サイト構成と情報提供の現状

KPLは13の主要ページで構成され、桐生市議会の12年分のデータを構造化・公開している。

| ページ | 機能 | 情報の深さ |
|--------|------|------------|
| `/` トップ | ハイライト・問いかけ型ナビゲーション | 広く浅い（入口） |
| `/sessions/` | 定例会・臨時会一覧（68会期分） | 年度別に整理 |
| `/sessions/[slug]` | 定例会詳細 | 議案テーブル・採決結果・投票マトリクス・分裂投票詳細 |
| `/council/` | 議員一覧（22名） | 写真・会派・委員会 |
| `/council/[slug]` | 議員詳細 | 投票サマリー・質問履歴・関連テーマ |
| `/finance` | 予算・財政ダッシュボード | 人口推移・予算内訳・経年比較・基金残高 |
| `/analysis` | 投票パターン分析 | ヒートマップ・会派結束度・反対票傾向 |
| `/topics/` | テーマ別一覧（40+テーマ） | スパークライン・件数 |
| `/topics/[tag]` | テーマ別タイムライン | 年度別エントリ・採決分布・関連テーマ |
| `/schedule` | 議会カレンダー | 本会議・委員会の日程 |
| `/search` | 全文検索（pagefind） | 横断検索 |
| `/guide` | 議会の仕組みガイド | 制度解説・用語集15語 |
| `/about` | サイト情報 | 方針・出典・技術・問い合わせ |

### 3.2 データの網羅性

| データ | 件数・範囲 | 粒度 |
|--------|-----------|------|
| 議員名簿 | 22名（+議長2） | 会派・委員会・当選回数・写真 |
| 定例会・臨時会 | 68会期（H24〜R7、12年分） | 議案番号・タイトル・採決結果・カテゴリ |
| 議案 | 554件 | 番号・タイトル・結果・議員別投票 |
| 投票記録 | 46会期分（H28〜R7、9年分） | 議案ごとに議員別個票（賛成/反対/欠席/議長） |
| 一般質問 | 43会期分（H27〜R7、10年分） | 議員・質問順序・テーマ・詳細（3階層） |
| AIタグ分類 | 600+エントリ | 1-3個のテーマタグ |
| 投票分析 | 議員×議員 | 一致率マトリクス・会派結束度 |
| 人口推移 | 101年分（1920〜2020） | 人口・世帯数 |
| 予算 | 当年度+3年経年（R3〜R7） | 歳入・歳出・基金・財政指標 |
| 議会日程 | 直近〜数ヶ月先 | 本会議・委員会・日時 |
| 新着情報 | リアルタイム（最新10件） | タイトル・URL・日付 |

**データの強み**: 議案×議員の投票個票レベルまで構造化されている点は、日本の地方議会モニタリングとしては極めて先進的。

**データの不足**:
- 一般質問への回答内容（市長・部長の答弁）が未取得
- 議案の内容詳細がない（タイトルと結果のみ、内容は公式リンク先依存）
- 過去の会派変更履歴が追跡されていない
- 議会だよりPDFが未構造化（OCR必要）
- 会議録が未取得（公式サイトがrobots.txtでクロール拒否）

### 3.3 現在の「気づき」促進機能

KPLには既にいくつかの分析・ハイライト機能が実装されている。

**実装済みの機能**:
- トップページ「データで見る桐生市の今」: 人口減少率、全会一致率、最多テーマ、1人あたり予算の4カード
- 問いかけ型ナビゲーション: 「税金は何に使われている？」等6つの問いから各ページへ導線
- 投票分析ページ: 会派結束度テーブル、議員間一致率ヒートマップ、反対票傾向
- 分裂投票の検出と表示: 全会一致でない議案を赤枠でハイライト
- テーマ別タイムライン: 40+テーマの時系列表示、スパークライン
- 財政指標の警告ライン: 経常収支比率・実質公債費比率に全国平均や要注意ラインを表示

**不足している機能**:
- 「この数字が何を意味するか」の解説が薄い（例: 全会一致率97.5%は良いのか?）
- 難易度の切り替え（やさしい/くわしいモード）がない
- テーマが「なぜ多く議論されているのか」の背景説明がない
- 定例会ごとの「今回何が起きたか」サマリーがない

### 3.4 現在の「行動」導線

**実装済み**:
- 全ページで公式サイト原文へのリンクを併記
- RSS フィード（`/rss.xml`）
- GitHub Issues での問い合わせ
- 議会カレンダー（日程表示）

**実装されていない**:
- 議会傍聴の方法・手続きへのリンク
- パブリックコメント募集案件のリスト
- 請願・陳情の提出方法
- 議員への連絡方法
- 議会中継の視聴リンク

### 3.5 背景説明の現状

**About ページに記載されている内容**:
- 「中立的なアーカイブサイト」「非公式」「個人事業主が運営」
- サイトの方針4ポイント（中立性・公式への入口・AI透明性・自動化）
- データの出典、技術情報、問い合わせ先

**記載されていない内容**:
- プロジェクト実施の動機（生活保護不正、入札妨害等の具体的背景）
- なぜ個人でこのプロジェクトを始めたのか
- 「市民が議会・市政を理解することの重要性」というメッセージ

これは大きなギャップである。訪問者は「きれいに整理されたデータベース」を見ることはできるが、「なぜこれが必要なのか」「なぜ今なのか」という文脈が伝わらない。

## 4. シビックテック事例研究: 先行者から学ぶ

### 4.1 海外の主要事例

#### TheyWorkForYou（英国 / mySociety、2004年〜）

世界初の市民向け議会モニタリングサイト。議員の発言・投票をデータベース化し、検索・アラート通知を提供する。

- **最も評価されている機能**: キーワードアラート。特定の言葉が議会で使われたらメール通知。「市民社会がパーラメントを監視するための不可欠なツール」と評されている
- **設計思想**: 議員の名前や選挙区から入る導線。「あなたの議員は議会で何を言っていますか？」
- **運営**: 非営利団体mySocietyが寄付＋助成金で運営。20年以上継続
- **KPLへの示唆**: アラート機能の実装価値。RSSベースで低コストに実現可能

#### TheyVoteForYou（オーストラリア / OpenAustralia Foundation、2006年〜）

KPLの投票分析機能と最も類似するモデル。

- **コンセプト**: 「あなたの代表はあなたのために投票していますか？」という問いを中心に設計
- **機能**: テーマ別の政策ポジション（賛成/反対の傾向）を自動算出。9,799件の投票データ、1,046名の議員を追跡
- **特徴**: オープンソース、APIでデータ再利用可能
- **KPLへの示唆**: テーマ別政策ポジションの可視化。KPLの投票データとタグ分類を組み合わせれば実装可能

#### g0v（台湾 / 零時政府、2012年〜）

分散型シビックテックコミュニティ。8,000人超のコントリビューター、40回以上のハッカソン。

- **象徴的アプローチ**: 「政府サイトのURLの "o" を "0" に置き換える」。つまり政府と同じ情報を、よりわかりやすく提供する
- **vTaiwan**: 市民・専門家・議員が法案を議論するプラットフォーム。約12の法律・規制に反映された実績
- **成果**: デジタル担当大臣（オードリー・タン）を輩出するなど、政策への直接的影響
- **KPLへの示唆**: 個人プロジェクトから始まり、コミュニティに発展するモデル。ただし台湾の文脈（市民社会の成熟度、デジタルリテラシー）とは異なる点に注意

#### Parliament Watch Thailand（WeVis）

KPLの成長モデルとして最も参考になる事例。

- **経緯**: ボランティアの若者チームから出発。5年間で段階的に機能を拡張
- **展開**: TheyWorkForUS（議員追跡）→ Law Watch（法案追跡）→ Promise Tracker（公約追跡）と段階的に拡大
- **特徴**: 「挫折→改善→学習」のサイクルを公開し、後続プロジェクトを鼓舞
- **KPLへの示唆**: 小規模チーム（個人）でも段階的拡張で大きな影響力を持てる

#### Qué Hacen Los Diputados（スペイン語圏）

「議員は何をしているか」を市民・ジャーナリスト・研究者にわかりやすく提供するプロジェクト。KPLと同様の問題意識を持つ。

### 4.2 日本国内の事例

#### みらい議会（チームみらい、2025年10月公開）

KPLと最も思想が近い国内プロジェクト。国会の法案をAIで「見える化」する。

- **特徴的な機能**:
  - 「やさしい」モードと「くわしい」モードの切り替え
  - 法案内の用語を選択するとAIが解説
  - 漢字にすべてルビを振り、子ども・外国人にもアクセシブル
- **設計思想**: 「情報は存在するが届かない」構造を変える
- **KPLへの示唆**: 難易度切り替え機能。AI要約を活用した「やさしい版」の自動生成

#### Code for Japan（2013年〜）

地方自治体へのICT人材短期派遣（フィールドラボ）を展開。新型コロナ対策サイトを1.5日で開発し約80自治体に展開した実績がある。オープンソース・オープンデータを基盤とした活動で、KPLが接点を持ちうるコミュニティ。

#### その他

- **選挙ドットコム / 選挙ウォッチ**: 候補者情報・選挙動向の集約。ただし議会活動の継続モニタリングとは異なる領域
- **地方議会会議録検索サイト（個人運営）**: 個人がボランティアで運営する会議録検索リンク集。地道だが持続している例

### 4.3 成功プロジェクトに共通するパターン

#### 情報の見せ方

| パターン | 具体例 | KPL対応状況 |
|----------|--------|-------------|
| 「問い」中心の設計 | 「あなたの議員は？」（TheyVoteForYou） | 部分的（問いかけナビはあるが深掘りが弱い） |
| 難易度の切り替え | やさしい/くわしい（みらい議会） | 未対応 |
| テーマ別政策ポジション | 議員ごとの賛成/反対傾向の自動算出 | 部分的（投票パターン分析あり） |
| キーワードアラート | 特定の言葉が議会で使われたら通知 | 未対応（RSSのみ） |
| アクセシビリティ | ルビ・多言語対応 | 未対応 |

#### 市民エンゲージメントを生む仕組み

1. **アラート・通知**: 受動的な市民を能動的に変える最も実績のある手法。TheyWorkForYouで「最も価値のある機能」と評価
2. **問いかけ型の導線**: 「あなたの議員は何をしていますか？」→ 受動的な情報閲覧を能動的な関心に変換
3. **データの再利用性**: API公開やオープンデータ化により、ジャーナリスト・研究者・他プロジェクトが自発的に活用し、エコシステムが生まれる
4. **ハッカソン・対面イベント**: g0vの成功要因。デジタルとリアルのハイブリッド

#### 持続可能な運営モデル

| モデル | 例 | 特徴 |
|--------|-----|------|
| 非営利団体＋寄付・助成金 | mySociety, OpenAustralia | 安定的だが助成金依存リスク |
| 分散型コミュニティ | g0v | 中核メンバーが変わっても存続。ただし形成に時間がかかる |
| 政党・政治団体主導 | みらい議会 | 資金安定だが中立性の課題 |
| 個人運営＋自動化 | KPL, 会議録検索サイト | コスト最小だがバス係数1 |
| 行政との協働 | Code for Japan | 行政側の理解が必要 |

**重要な知見**: シビックテック全体として「過去10年で意味のあるスケールや持続性を達成したものはごくわずか」という現実がある。多くがビジネスモデルなく立ち上げられ、収益を確保できていない。KPLの自動化による低コスト運営（月数百円）は、この文脈では大きなアドバンテージである。

## 5. ユーザーセグメントとニーズ

### 5.1 想定利用者の分類

| セグメント | 推定規模 | ニーズ | 行動パターン | KPLへの期待 |
|---|---|---|---|---|
| **一般市民（関心高）** | 500-1,000人 | 議案の背景理解、議員の実績評価 | 定例会前後にアクセス、SNSで共有 | 投票記録・議員比較が核心価値 |
| **一般市民（関心中）** | 3,000-5,000人 | 「自分に関係ある話」だけ知りたい | 大型事案（庁舎・図書館）で単発アクセス | テーマ別タイムラインが入口 |
| **一般市民（関心低）** | 90,000人以上 | そもそもアクセスしない | SNSで流れてきたら見る程度 | KPLの直接ターゲットではない |
| **地元メディア** | 5-10社 | ファクトの裏取り、経年データ | 記事執筆時にデータソースとして利用 | 構造化データの一次資料価値 |
| **市議会議員・行政関係者** | 22議員+数十名 | 自分の実績の可視化/他議員の動向 | 定期的にチェック（見え方を気にする） | 味方にも脅威にもなる最重要ステークホルダー |
| **市民活動家・NPO** | 50-100人 | 政策提言のエビデンス | 特定テーマを深掘り、資料として引用 | テーマ別横断検索、データダウンロード |
| **研究者** | 数名 | 地方議会研究の事例 | 不定期、データセットとして利用 | API的アクセス、メタデータの充実 |

### 5.2 戦略的含意

**最重要ターゲットは「関心中」の一般市民（推定3,000-5,000人）。**

- 「関心高」は放っておいてもたどり着く
- 「関心低」はKPL単体ではリーチ不能
- 「関心中」を「関心高」に引き上げることが最大のインパクト

そのためのキーは **「自分ごと化」**。議案番号ではなく「図書館がどうなるか」「ゴミ収集がどう変わるか」という生活者目線での情報提示が必要。

**副次的に重要なのは地元メディアと市議会議員。** メディアがKPLを引用すれば、間接的に「関心低」層にもリーチできる。議員がKPLを「敵」ではなく「自分の活動を可視化するツール」と認識すれば、プロジェクトの持続性が大きく高まる。

## 6. サイトの方向性: 中立アーカイブの進化形

### 6.1 「中立アーカイブ」か「市民参加促進プラットフォーム」か

**結論: 「中立アーカイブ」を堅持すべき。ただし「行動のための素材提供」までは中立の範囲内。**

#### 中立を堅持すべき理由

1. **信頼性が最大の資産**: 中立性を失えば「あのサイトは偏っている」と言われた瞬間に価値がゼロになる。議員や行政からの信頼も同時に失う。個人運営であるがゆえに、組織的な信頼の裏付けがない分、中立性の実績だけが信頼の源泉

2. **市民参加促進は別のアクターがやるべき仕事**: KPLが提供した素材を使って、市民団体やメディアが「参加促進」を行うのが健全な役割分担。KPLが両方やると、データの客観性まで疑われる

3. **Layer 1-3は完全に中立の範囲内で実現可能**: 「全会一致でない議案をハイライトする」「予算の前年比を可視化する」は事実の提示であり、意見ではない

### 6.2 中立の範囲内でできること・できないこと

**KPLがやるべきこと**:
- 情報の構造化・可視化（Layer 1）
- 事実に基づく文脈付与（Layer 2）: 経年比較、類似自治体比較、背景リンク
- 客観基準による注目点の検出（Layer 3）: 分裂投票、予算異常値、パターン変化
- 行動の素材提供（Layer 4の一部）: 傍聴ガイド、パブコメ案内

**KPLがやるべきでないこと**:
- 政策の評価・批判
- 議員のランキング・スコアリング
- 特定の立場からの市民動員
- 「この議案について議員に聞いてみよう」的な誘導
- 公式サイトの代替を目指すこと

### 6.3 4層モデルによる現状と目標

```
                        現状        目標（6ヶ月後）
Layer 1: 情報アクセス   ████████████  ████████████  （維持・微調整）
Layer 2: 文脈と解説     ████░░░░░░░░  ████████░░░░  （重点強化）
Layer 3: 気づきの促進   ██░░░░░░░░░░  ██████░░░░░░  （選択的に強化）
Layer 4: 行動への導線   █░░░░░░░░░░░  ████░░░░░░░░  （最低限の導線設置）
```

### 6.4 単一サイトか複数サイトか

**結論: 当面は一つのサイトで十分。**

- 現在の情報量は一つのサイトで管理可能
- 複数サイトは運用負荷が個人の限界を超える
- 将来「kiryu.life（暮らし情報）」を分離する構想があるが、KPL本体のLayer 2-3が充実してからで遅くない
- 分離するなら**データ種別（議会 vs 暮らし）**で分けるのが自然。機能別（アーカイブ vs 参加促進）で分けるのは不適切

## 7. 担保すべき情報: 何を深め、何を追加するか

### 7.1 情報の各側面の評価と方針

| 側面 | 現状 | 方針 |
|------|------|------|
| **網羅性** | ◎ 12年分554議案・全投票個票・全質問 | 維持。答弁内容は中期課題 |
| **詳細度** | ○ 投票粒度は十分 | 議案の内容要約をAIで追加検討 |
| **解説** | △ 用語集15語、ガイドあり | ツールチップ化、「何を意味するか」の説明追加 |
| **閲覧性** | ○ ヒートマップ等の可視化充実 | 難易度切替（やさしい/くわしい）の検討 |
| **文脈** | △ 経年比較は一部あり | 前回定例会との比較、異常値検出を追加 |
| **背景説明** | ✗ 動機の記載なし | About ページの充実（最優先） |
| **行動導線** | ✗ 傍聴・パブコメ等なし | 静的ページとして追加 |

### 7.2 Layer 2（文脈と解説）の具体施策

| 施策 | 難易度 | 効果 | 中立性リスク |
|------|--------|------|-------------|
| **経年比較の自動生成**（前年同議案との比較、予算の前年比） | 中 | 高 | 低（数値事実） |
| **議案の「背景情報」リンク集**（関連する過去議案、新聞記事URL） | 中 | 高 | 低（リンクのみ） |
| **用語解説の自動付与**（議会用語にツールチップ） | 中 | 高 | 低 |
| **定例会サマリーの自動生成**（AI要約、議案数・可否の統計） | 中 | 高 | 中（要約の偏りに注意） |
| **類似自治体との比較**（人口規模が近い自治体の公開データ） | 高 | 高 | 低（数値事実） |
| **投票パターンの変化検出**（「この議員は今期、反対票が増えた」） | 低 | 中 | 中（解釈を誘導しうる） |

### 7.3 Layer 3（気づきの促進）の具体施策

| 施策 | 難易度 | 効果 | 中立性リスク |
|------|--------|------|-------------|
| **「注目の議案」自動検出**（全会一致でない議案のハイライト） | 低 | 高 | 低（客観基準） |
| **予算の異常値検出**（前年比±20%以上の項目を自動フラグ） | 中 | 高 | 低（客観基準） |
| **議員活動スコアリング** | 中 | 高 | **高（実装しない）** |

**Layer 3の原則**: 「異常値」を検出する基準は必ず**機械的・客観的**なものに限定する。「全会一致でない」「前年比20%超」など、誰が見ても同じ結果になる基準のみ使う。基準はサイト上に必ず明記する。

### 7.4 Layer 4（行動への導線）の具体施策

| 施策 | 中立性との両立 |
|------|---------------|
| 「傍聴のしかた」ページ（日程・場所・手続きの事実情報） | **完全に両立**（公式情報の再掲） |
| 議会事務局・各課の連絡先リンク | **完全に両立** |
| パブリックコメント募集中案件のリスト | **完全に両立**（公式情報の集約） |
| 情報公開請求の手続きガイド | **完全に両立** |
| 「この議案について議員に聞いてみよう」的な誘導 | **リスクあり（やらない）** |

## 8. 気づきと行動への仕組み

### 8.1 ファネル設計

KPLの目標は、「情報は存在するが見えにくい」状態を解消し、市民が段階的に市政への関心を深められる導線を設計することである。

```
┌─────────────────────────────────────────────┐
│ 関心低の市民（90,000人+）                    │
│   KPLの直接ターゲットではない                │
│   → SNSで「定例会サマリー」が流れてくる      │
└────────────────────┬────────────────────────┘
                     ▼
┌─────────────────────────────────────────────┐
│ 関心中の市民（3,000-5,000人）★最重要★       │
│   「自分に関係ある話」だけ知りたい            │
│   → テーマ別タイムラインで「自分ごと化」      │
│   → 図書館・ゴミ収集・福祉 = 生活者目線      │
└────────────────────┬────────────────────────┘
                     ▼
┌─────────────────────────────────────────────┐
│ 関心高の市民（500-1,000人）                  │
│   議案の詳細・投票パターンまで追いたい        │
│   → 投票マトリクス・会派分析・経年比較        │
│   → キーワードアラートで継続的にフォロー      │
└────────────────────┬────────────────────────┘
                     ▼
┌─────────────────────────────────────────────┐
│ 行動する市民                                 │
│   → 傍聴ガイド・パブコメ案内で次のステップへ  │
│   → データをSNSでシェア、メディアに提供       │
└─────────────────────────────────────────────┘
```

### 8.2 「関心中」→「関心高」の引き上げ施策

これがKPLにとって最も投資対効果の高い領域である。

#### 定例会サマリーの自動生成（最重要施策）

「今回の定例会で何が起きたか」を1ページで把握できるようにする。

- **内容**: 議案数の分類別集計、全会一致でない議案のハイライト、注目テーマ、質問の要約
- **実装**: 既存データ + Claude Haiku による要約で自動生成可能
- **効果**: SNSでシェアされやすい。メディアにも引用されやすい。「関心中」市民の入口として最適
- **運用負荷**: 自動生成なのでほぼゼロ
- **タイミング**: 次の定例会（6月）に合わせて準備

#### テーマ別「自分ごと化」の強化

- 既存のテーマ別タイムラインに、生活者目線の導入文を追加
- 例: 「図書館」→「桐生市立図書館の建て替えに関する議論の経緯」
- 例: 「生活保護」→「桐生市の生活保護制度をめぐる議論」

#### キーワードアラート / メール通知

- TheyWorkForYou で最も価値があると評価されている機能
- 「生活保護」「新庁舎」「図書館」など関心テーマを設定し、関連議案が出たら通知
- 技術的にはRSS基盤 + Buttondown等のメール配信サービスで低コスト実装可能

### 8.3 行動への橋渡し

KPL自体が行動プラットフォームになる必要はない。しかし「行動への橋渡し」は中立の範囲内で可能であり、かつ重要である。

**追加すべきページ**:

1. **「市政に関わるには」ガイドページ**
   - 議会傍聴の方法（場所・手続き・日程へのリンク）
   - パブリックコメントの出し方（現在募集中の案件リスト）
   - 情報公開請求の手続き
   - 市民相談窓口の一覧
   - すべて公式情報の再掲・集約であり、KPLの意見は含めない

2. **データの共有・再利用の促進**
   - SNSシェア用OGP画像の最適化（既に実装済み、さらに改善）
   - データのCSV/JSONダウンロード機能（ジャーナリスト・研究者向け）
   - 将来的にはAPIの公開

## 9. リスクと留意点

### 9.1 中立性の担保と「気づきの促進」の両立

これはKPLにとって最も重要な設計判断であり、常に意識すべき原則である。

**実践的なガイドライン**:

| 原則 | 具体例 |
|------|--------|
| 「なぜ」を書かない、「何が」だけを示す | 「賛否が分かれた」は事実。「問題のある議案」は評価 |
| 基準を明示する | 「注目」の定義をサイト上に記載（例: 「全会一致でなかった議案を自動表示」） |
| AI要約には原文リンクを必ず併記 | 現在の方針を継続 |
| 議員スコアリング・ランキングは実装しない | 投票記録の提示と、ランキング化には質的な違いがある |
| 解説と意見を峻別する | 用語解説はOK。「この予算配分は問題だ」はNG |

### 9.2 行政・議会との関係性

**最大のリスクは「敵視される」こと。最大の機会は「公式が参照する」こと。**

| 施策 | 詳細 |
|------|------|
| **先手を打つ** | サイト公開時に議会事務局に一報を入れる。「公開情報を整理しているサイトです」程度。奇襲と受け取られるのが最悪のパターン |
| **データの出典を徹底明記** | すべてのページに「出典: 桐生市議会公式サイト」を記載。公式サイトへのリンクを目立つ位置に配置（既に実施中、さらに強調してよい） |
| **誤りの指摘に即応** | 議員や行政から「このデータは間違っている」と指摘された場合、迅速に確認・修正する姿勢を示す。連絡先を明記 |
| **「敵ではない」シグナル** | Aboutページで「公式情報への入口」という位置づけを明確にする（現在の方針を継続） |

### 9.3 情報の正確性と更新頻度

| リスク | 対策 |
|--------|------|
| スクレイピング対象サイトの構造変更 | CIの失敗通知を確実に受け取る体制（GitHub Actions + 自動PRで概ね対応済み） |
| データの欠損・不整合 | 定例会ごとの手動スポットチェック（年4回、各30分程度） |
| AI要約の品質劣化 | 要約結果のサンプリング確認を定例会ごとに実施 |
| 公式サイトのrobots.txt変更 | 半年に1回チェック |

### 9.4 法的リスク

- 公開情報の再掲載は著作権法上の問題は低い。ただし**データベースの著作物性**（議案リストの「選択・配列の創作性」）が理論上の争点になりうる。実務上のリスクは極めて低いが認識はしておく
- 議員の顔写真を使う場合は肖像権に注意。公式サイトの写真をそのまま転載するのではなく、公式サイトへのリンクにとどめるのが安全

### 9.5 持続可能性

- **バス係数1**: 個人運営のため、運営者が離脱すればプロジェクトが停止する
- **対策（段階的）**:
  1. 短期: 自動化の徹底により、運用の手間を最小化（現在の方針を継続）
  2. 中期: 地元メディア・市民団体との連携。データ提供という形での協力関係
  3. 中期: Code for Gunma / Code for Kiryu 等の地域コミュニティとの接点づくり
  4. 長期: データAPIの公開による外部エコシステムの促進
  5. 長期: ドキュメント整備により、他の人がフォーク・運営を引き継げる状態にする

## 10. ロードマップ: 具体的な次のステップ

### 6ヶ月スパンの優先順位

| 優先度 | 施策 | 工数 | 時期 | 理由 |
|--------|------|------|------|------|
| **P0** | Aboutページに動機・背景を追加 | 小 | 今すぐ | プロジェクトの「なぜ」を伝える最低限の対応 |
| **P0** | 全会一致でない議案のハイライト強化 | 小 | 今すぐ | 既存データから実装可能、効果大 |
| **P1** | 定例会サマリー自動生成 | 中 | 6月定例会前 | 「関心中」市民の最大の入口。自動化で運用負荷なし |
| **P1** | 傍聴ガイド・パブコメ案内ページ | 小 | 6月定例会前 | 一度作れば更新頻度低い |
| **P1** | 用語ツールチップ | 中 | 6月定例会前 | 「関心中」市民の離脱防止に直結 |
| **P2** | 「やさしい解説」モード（AI要約活用） | 中 | 夏 | みらい議会に学ぶアクセシビリティ向上 |
| **P2** | キーワードアラート（RSS→メール） | 中 | 夏 | 最も実績のあるエンゲージメント手法 |
| **P2** | 予算異常値検出 | 中 | 夏（予算データ更新時） | 年1-2回だが印象的 |
| **P3** | データダウンロード・API公開 | 中 | 秋 | エコシステム形成の基盤 |
| **P3** | 類似自治体比較 | 高 | 秋以降 | 効果高いがデータ収集コストも高い |
| 保留 | 会議録テキスト化 | 高 | 未定 | robots.txt問題・OCR問題で技術的障壁大 |

### 最も重要な一手

**次の定例会（6月）に合わせて「定例会サマリー」ページを自動生成する仕組みを作ること。**

「今回の定例会で何が起きたか」を1ページで把握できるようにする。これが「関心中」市民にとっての最大の入口になる。SNSでシェアされやすく、メディアにも引用されやすい形式であり、自動生成なので運用負荷もほぼゼロ。
